The Smartwatch App

My smart watch app is a simple forecasting app for windsurfing. It will provide on-demand forecasts, alerts for upcoming desirable weather and recommendations for equipment selection.

Why a windsurf and forcast app? I realise that this is an app for a very niche market. However I see this as a prototype for an app that can have a much broader appeal. There are many popular wind and water sports that rely on weather and ocean conditions (surfing, kite-boarding, wind-surfing, kayaking, paddle boarding, fishing, snorkelling and diving to name a few) and my proposed app could be extended to include these sports with relative ease. I am a novice windsurfer and so I understand some of these forecasting tools and I use them fairly regularly that is why I have chosen to use this sport as a proof of concept for this app. If this idea were to be applied to surfing for example the alerts could be based on wave height forecasting, for fishing it may be based on tide information with a bias from the weather and wind direction possibly even phases of the moon.

Windsurfing is a sport that relies on having the appropriate weather for your corresponding skill level and also choosing the right equipment for the given weather. The app will periodically retrieve weather forecast information from an online weather API and store it locally. Initially this will be simulated but the aim is to have it call a real API, such as the Magic Seaweed API, in the final version. With the weather information now in the app the user will be able to simply browse the forecast like any other weather app or they will be able to have personalised alerts set up that will let them know of upcoming favourable weather.

The app will also provide gear recommendations for any given forecast. It is quite common for windsurfers to own a quiver of different sail sizes (usually 3 - 6 sails) to cover a range of weather conditions, and to match their skill level and weight. A lot of windsurfers use online calculators to make these decisions such as the popular calculator on James’ Blog. I plan to use the formula from this calculator to make the sail recommendations in my app. Sails are measured in square meters and range between about 3.0 m2 and 9.5 m2. For example a small sail for very strong winds may may be 4.5 m2 and the weather for such a sail is colloquially termed “4.5 weather”. For my app I will use this two digit decimal number only (without the unit reference) to refer to the sail size as is often done when speaking about windsurf sails.

Alerts and equipment recommendations will be based on user set preferences. Initially I had thought that fewer user settings would be better as it would make the app simpler overall (not only simpler to code but simpler for the user as well) however with fewer personalisation settings the more likely it is to give unwanted alerts. Some balance needs to be reached between the number of personalisation settings and the simplicity of the app. The personalisation settings will include some combination of the following:

Skill Level:
This would be a single choose from 3 or 4 different options such as “novice” or “experienced”.
Wind Preferences:
This will be a multi-choice set of options such as “gentle breeze sailing”, “brisk winds” and “I’ll be out there in hurricane force”.
Weather Preferences:
This will also be multi-choice set of options such as “sunny”, “warm”, “cold” or “anything goes”.
User Weight:
Sail selection is greatly affected by rider weight.
Sails Owned:
If the user can add the sail sizes they own to the app then the app can then make a direct recommendation of which sail to use.
Location:
Weather reports will be based on a specific location. For proof of concept I plan to hard code the location in but a final app my include the ability to set up alerts for multiple locations. For example I might go windsurfing in Dunedin harbour on any slightly windy condition because I live nearby, but I might drive up or down the coast to specific locations but only if the forecast conditions reach some specific threshold.

Version Control

Little research it is required to see that Git is the most widespread version control system in use today. This is backed up by a 2014 survey of eclipse users which showed that 33.3% of the developers surveyed used Git followed by 30.7% of developers who used Subversion. The survey also noted a continued downward trend of Subversion users. Looking through job advertisements on trademe.co.nz and seek.co.nz also demonstrates the widespread adoption of Git, “understanding Git” appears to be a common skill requirement for advertised software developer positions.

For these reasons, and the fact that I already have some limited experience using Git from the Terminal, Git will be the version control system of choice for this project.

The main Git repository hosting services available are GitHub and GitBucket.

GitBucket offers free accounts with unlimited private repositories but the free service is limited in the number of collaborators you can have on a project (5 being the limit). GitHub on the other hand offers free accounts with unlimited collaborators but with the limitation of no private repositories.

As far as support goes it appears that GitHub is more widely used on the web. Viewing multiple youtube tutorials on how to use Git it appears that GitHub is the most widely recommended service. Being so widely used it has the additional advantage of have an vast amount of supporting material online. A quick google search of “how to do XYZ on GitHub” brings up resources and documentation form the official site, questions and answers from StackOverflow, guides on blogs such as lifehacker and sites such as wikihow

In addition to the widespread support for GitHub they also provide coupons for free “Micro Accounts” (usually $7 per month) with private repositories for tertiary students as part of a student pack. The student pack also comes with free access to a mix of other services (hosting, text editors, DNS management etc.) that are not normally free.

GitHub will the the repository used for this project.

Editors and IDEs

Myself and another group member (Matt Wynyard who later left the course) began exploring Editors and IDEs very early on in the assignment. At the time I was just starting to become familiar with using Sublime Text for writing HTML and CSS code and using the Terminal to operate Git and push/pull from GitHub. Matt was very keen on everyone in our group using a single IDE for consistency, his suggestion at the time was NetBeans. I installed and tested NetBeans on my computer for a couple of weeks just writing HTML & CSS while using the built in Git & GitHub features to commit, push, pull, branch and merge.

While JetBeans provided all the functionality that we needed it also came bundled with a lot more features that weren’t needed, it seemed overly complex to start very simple new projects. After some testing with the GitHub integration in JetBeans (intentionally branching, merging, deleting branches etc.) I managed to ‘break’ the the repository I was testing it on and no matter what I tried I was unable to repair the repository and I would repeatedly get indecipherable error messages from the JetBeans application. I had not experienced this issue while exploring the functionality of Git and GitHub from the Terminal.

My personal preference is to stick with the Sublime Text editor in conjunction with the Terminal for all my coding work and version control. Even if I were still in a group at this stage I would see no reason for everyone to be using the same IDE or text editor if we were all working from a remote repository.

Sublime Text itself is seemingly simple but deceptively powerful text editor with extendable functionality (using plugins). The editor is multi-platform and comes at a fairly reasonable price of $70 and offers an extensive trial period. To test the ease and practicality of it’s claimed extendability I decided to see how easy it would be to to make Sublime Text recognise Sass files and syntax.

  • First I needed to install Sublime Package Control which is recommended for adding on extra packages. To do this I followed the online instructions of opening the Sublime Text console and pasting in the instruction code to install. This took less than 2 minutes to complete.
  • The next step was to open the Command Palette in Sublime Text then click on “Package Control: Install Package”. This opened a search bar, I entered “Sass” and clicked on the first and most appropriate option that brings Sass support to Sublime text and the install took less than a minute.

Overall it took me less than 5 minutes to add extra functionality to Sublime Text that I was interest in. It worked flawlessly and my version of Sublime Text now recognises any and all Sass files I open.

In addition to Sublime Text and Teminal I discovered the website CodePen that appears to be very useful for working with HTML, CSS & Javascript. CodePen is marketed as a “Front end developer playground” and runs entirely within the web browser. When you log into CodePen the first thing you do is create a new ‘pen’ which consists of 4 panels: a panel each for HTML code, CSS code and Javascript code, and a fourth panel that displays the rendered code. This would be very useful for running small pieces of code (for example Javascript functions or CSS styles) and seeing how they function in the real world, or in isolation from a complete app, with instant feedback. A good place to test and brainstorm small ideas before inserting them into the final code.

Debugging Tools

Having some experience with writing HTML and CSS I am familiar with using the Firebug web development tool for inspecting HTML and making changes to live CSS. While proven useful for HTML & CSS inspection Firebug is also advertised as “The most advanced JavaScript debugger available for any browser”.

As a Javascript debugging tool Firebug allows the ability to set breakpoints within the Javascript at the click of a button and then allows stepping through the code one step at a time. Firebug also allows the ability to inspect variables, objects and even the call stack while the script is paused.

At this stage, with my current understanding of Javascript, the Firebug tool seems more than adequate as a debugging tool for the anticipated size and complexity of my emulator & smart watch application.

I am familiar with unit testing Java and would like to include some unit testing for my Javascript. QUnit appears to be one of more popular unit testing frameworks for Javascript (although there is no obvious internet-favourite of this category) and there are online tutorials that provide useful introductions on how to integrate QUnit tests into Javascript code.
