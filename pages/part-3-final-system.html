Introduction

The emulator and app prototype are now presented in their final state. The emulator has seen multiple improvements over what was put forward in Assignment 2 including a time advancing function, a more logical set of buttons and a way of preventing nonsensical button input form the user. The app prototype now demonstrates alerts through the use of the time advancing feature. The app also processes raw weather data files into a locally stored data structure.

The ability to be able to change the settings of the app has been removed from the original proposal of Assignment 1 however all other features remain intact in one form or another and the app prototype functions as planned.

Not working in a group has been a big obstacle of this assignment but despite this I have learnt a lot of new tools, languages and techniques.

Final Emulator

The final emulator has only undergone minor changes compared to the emulator presented in Assignment 2 which reflects the near complete state the emulator was in for the Assignment 2 submission. The three most significant changes to the emulator are: improved button layout and control, a time advancing feature and a thorough overhaul of code documentation.

All nonessential buttons (i.e. buttons that have no specific use for my app) that were present in Assignment 2 have been removed and a number of extra buttons have been added, namely: "Reset", "Sleep", "Wake" and "Advance Time". All buttons now also have a dynamic "active" and "inactive" state to reflect if they are able to be used in the current context. For example when the page is first loaded the only button that can be pressed is "Activate App" and once the app is activated the "Activate App" button can no longer be used. This dynamic button state greatly improves the user experience and prevents nonsensical input (such as trying to activate the app when the app is already activated).

A simple time advancing feature has been added to the emulator which consists of a variable to track how many units of time the emulator has been advanced (note: the units are purposefully abstract as my app interprets them as days but another app may interpret them as minutes or hours) and an "advance" instruction that is sent to the instruction queue.

The last major change to the emulator is a major overhaul to the documentation of the code to make it easy to follow and understand for other users.

App Prototype

What it does & How to use it

My smartwatch application prototype is a weather watching tool for windsurfers. It provides current weather information, weather forecasts and sail size recommendations for the weather conditions. The app can be used actively to check forecasts and conditions or it can be used in a more passive way where it can be set to alert the user when some favourable conditions are forecast in the coming days.

For this prototype I have hardcoded the user settings and for the purposes of demonstration use it is worth noting that the settings for preferred conditions has been set to anything between "5.2" and "7.0" conditions. What this means is that the user doesn't have a sail size smaller than 5.2 meters squared or a sail larger than 7.0 meters squared.

Home Screen

When the app is opened the user is immediately taken to an easy-to-read information-dense home screen with the following features:

  • A very prominent sail size recommendation for the current weather (note that windsurfers will often refer to the weather conditions in terms of the sail size e.g. "It's looking like a good 4.5 breeze out there today"). This number is often the only thing someone might want to know and it would normally involve them checking the windspeed and then making a some sort of calculation to get this number (either with a calculator or mental rule of thumb). As such this number is given prominence in the app home screen.
  • A written and graphical weather description
  • Actual predicted / measured weather conditions (wind speed in knots, wind direction, and temperature)
  • Warnings of severe weather conditions. Although no description of the warning specifics are provided to the user the importance of raising such warnings is of great importance for the safety of the user.
  • Current location and current day.
  • A forecast icon
  • A settings icon

From the home screen a typical user may want to check the upcoming forecast or simply sleep the app and go about their day.

Forecast Screen

When the user taps on the forecast icon on the home screen (the calendar on the bottom left) they are taken to the forecast screen which shows a condensed forecast of the upcoming four days. Each days forecast contains the following information:

  • Sail size recommendation for the day, again this being the most prominent element of the forecast.
  • What day the forecast is for
  • An icon depicting the forecasted weather (sun, clouds, rain, wind etc.)
  • Forecasted wind range in knots and wind direction
  • Forecasted temperature

While on the forecast screen the user can take two different actions. They can either "Swipe Left" which takes them back to the home screen or they can tap an a particular day to get a full screen display of that days forecast. Any action taken on the full screen forecast view will take the user back to the home page.

Alerts

Each new day the forecast updates in the background during which new forecast data is obtained and processed in the app. The app then checks the next four days for any forecast that is deemed favourable according to the users settings and will alert the user of the first (and only the first) favourable forecast that it identifies.

An alert consists of a screen outlining the forecast for the day in question and a series of beeps which are written to the page immediately below the emulator screen.

Any input from the user on an alert screen takes them back to the home screen or if the watch was in "Sleep Mode" then any input returns the watch to the sleep state.

Example Use

Below is a series of actions that can be performed to demonstrate the features of the app. Note that all actions must be taken - and in the correct order - to get the described result.

  • Reload the web page to ensure a fresh start.
  • Click the "Activate App" button to load the emulator and app. You are immediately taken to the home screen where you can see todays weather.
  • Tap the "Calander Icon" on the screen to see the forecast for the next four days.
  • Tap on one of the four forecast days to be taken to a full screen view of that day.
  • Click the "Swipe Left / Back" button to return to the app home screen.
  • Click the "Advance 1-Day" button once to see the app change to the next day.
  • Click the "Sleep" button to sleep the watch.
  • Click the "Advance 1-Day" button once to advance to the next day, the watch will wake and alert the user of some new upcoming favourable forecast.
  • Tap the screen anywhere to dismiss the alert and return the watch to it's sleep state.

Tools Used During Construction

Below is an overview of the main tools that were used while creating the emulator and app prototype.

Sublime Text 3

All code for this project was written in Sublime Text 3 editor which I have immensely enjoyed. Sublime was easy to pick up for a first time user because on the surface it appears to be a very simple text editor with only basic features. However through continued use I discovered many powerful and useful hidden features such as simultaneous editing or the project manager which allows quick switching between projects which preserves all the windows and displays open at the time of exiting.

Many of these features make some of the more mundane developing tasks a lot easier, for example I like to write my reports in a word processor and then paste the completed report into the HTML document and format from there. Sublime makes wrapping text in the correct HTML tags and replacing escape characters a relatively short and easy task which can otherwise take a long time in other editors.

Sublime Text also has a package manager that makes it easy to find and install third party plug-ins to make certain tasks easier. One example I used was the "AutoPrefixer" plugin. AutoPrefixer parses CSS and adds vendor prefixes to CSS rules. I prefer to use CSS flexbox for the layout of certain HTML elements such as the grid of buttons. After getting my layout how I want it in one browser I am able to run AutoPrefixer to make this layout the most compatible it can be with other browsers.

jQuery

Although I barely scratched the surface of what the jQuery library is capable of I found it to be incredibly useful for managing the interface between the user and the emulator controls. All of the buttons actions and dynamic states are controlled through the use of jQuery.

Testing

I used QUnit to carry out unit testing on my emulator and app prototype. Initially I chose QUnit because of its wide adoption and similarities to Java's JUnit, a tool I have used before. However I struggled a little with using QUnit. I encountered multiple issues with trying to make complicated tests on the state of my emulator and app.

QUnit's documentation describes methods for testing user behaviour through jQueries 'trigger' function but no matter how many different ways I tried I simply couldn't get it to work. I read multiple cookbooks, tutorials and how-to's on using QUnit but I had more trouble managing this than any other part of the project.

Testing is an area I have a lot of interest in so my struggles were not due to a lack of enthusiasm. Perhaps if I was in a group with other members my workload would have been more managable and I could have dedicated more time to develop a better understanding of QUnit. Or perhaps in a group I could have afforded the time to explore other testing options.

Due to the difficulties I had with QUnit I made a lot of use of the 'console.log' and 'alert' functions in JavaScript during development. When developing functions I would place these throughout to keep tabs on what was happening.

I also had Firebug open whenever I was working on my emulator and app. Firebug was useful for catching errors in my script as well as monitoring variables and the DOM.

Git & GitHub

I recognised early on that Git is a tool that is commonly used in the real world of software development so I was quick to adopt and learn this tool. Although this project never required a complex tree of different branches in my Git repository I did make use Git and GitHub throughout the project and feel that I will continue to build on this knowledge in the future.

Not Working in a Group

I was originally in a group of four but left the group prior to the submission of Assignment 1 due to communication issues. As such I have no group dynamics to reflect upon. Instead I have decided to reflect on the decision I made and the consequences it had on my work.

Working solo on this assignment had both advantages and difficulties. I would have preferred to have progressed through this project in a group. I enjoy working in groups and I tend to learn a lot from other people either through asking them questions or simply reading through their code and thinking about how their processes and logic differs from my own. However I still feel I made the right choice when leaving my group. Working on a project with other people (be it a group of students at University or a team at the workplace) where no-one wants to help others or communicate despite repeated requests is not healthy and is very likely to be at the detriment of the project.

Advantages of working solo:

The main advantage of working solo on a project like this has been the freedom to change direction without the need to consult others. There has also been no need to fit my code in to someone else’s structure or framework, everything has been of my own design and this allows a lot of freedom.

Working solo also affords one to have a much looser schedule and deadlines, I don’t need to desperately finish some features so other group members can start using them to write their own code.

Lastly working solo allows much tighter controls on consistency. Consistency in style, design and quality. I don’t need to pick up the pieces of someone not pulling their weight and there is no need to try and mash together two quality works that look completely different.

Disadvantages of working solo:

The most immediate disadvantage of leaving my group is that the instant I left my workload for this project increased fourfold! This massive increase in workload has taken its toll on the quality of the work I like to produce and the main area this can be seen is my unit testing. As discussed in the tools section testing is a topic that I am genuinely interested in but I struggled with using QUnit for this project. Given more time I may have either ironed out these issues to write deeper and more meaningful tests or I may have even switched to a different testing suite. Given teammates to work with I may have been able to sit down with someone one afternoon and discuss my issues with QUnit to get better insight on what I was doing wrong.

Working solo also brings the disadvantage of not being able to discuss and flesh out ideas with other group members. The way my emulator delivers instructions to the app through the instruction queue works fine but is something I spent a lot of time changing and re-visiting. I believe this part of my emulator could be much improved by simply sitting down with someone else and discussing possible alternative approaches.

Not having another "human validator" is another disadvantage of not working in a group. All to often validation tools will tell you that your code makes sense but another pair of human eyes may be able to see a problem with the logic of the code you have written.
